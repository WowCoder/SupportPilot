## Context

当前 RAG 系统采用单一检索策略（向量检索 + 重排序），存在以下局限：
- 无法处理复杂查询（如"对比 A 和 B 的异同"、"列出 2024 年发布的feature"）
- 不支持多跳推理（如"X 的作者在 Y 公司工作，该公司有哪些产品"）
- 检索策略固定，无法根据查询类型动态调整
- 代码耦合度高：RAGUtils 类承担过多职责，难以扩展和维护

**保留的现有模块**：
- 聊天记忆系统：独立模块，负责对话历史管理，Agent 可通过工具调用获取上下文
- Small-to-Big 检索：已验证有效的检索策略，作为工具之一保留

## Goals / Non-Goals

**Goals:**
- 引入 Agentic RAG 架构，支持复杂查询的自主规划和迭代检索
- 实现查询路由器，自动识别简单/复杂查询并分发
- 模块化重构代码，提升可维护性和可扩展性
- 保持现有 API 兼容，平滑迁移

**Non-Goals:**
- 不保留旧代码并行运行（直接切换，保持代码简洁）
- 不引入新的向量数据库（继续使用 ChromaDB）
- 不改变 embedding 模型（继续使用 all-MiniLM-L6-v2）
- 不涉及前端 UI 改动（仅后端架构优化）

## Decisions

### 1. Agent 编排框架：LangGraph vs 自定义状态机

**决策**: 使用 LangGraph

**理由**:
- LangGraph 提供成熟的状态机编排，支持循环、条件分支、并行执行
- 与 LangChain 生态无缝集成，工具定义简单
- 支持可视化调试和状态检查点
- 社区活跃，文档完善

**替代方案**: 自定义状态机
- 优点：轻量、可控
- 缺点：需要重复造轮子，维护成本高

### 2. Agent 架构：单 Agent vs 多 Agent 协作

**决策**: 单 Agent + 多工具模式

**理由**:
- 当前场景复杂度适中，单 Agent 足够处理
- 多工具模式灵活性高，可按需组合
- 性能开销小，响应速度快

**未来扩展**: 如需处理更复杂场景（如多轮对话、多角色协作），可升级到多 Agent 架构

### 3. 代码组织：按功能模块 vs 按层次分层

**决策**: 按功能模块组织

```
rag/
├── core/           # 核心接口和工具基类
├── tools/          # 检索工具实现
├── agents/         # Agent 定义和状态机
└── utils/          # 工具函数
```

**理由**:
- 功能内聚，每个模块职责清晰
- 易于测试和替换
- 符合当前团队习惯

### 4. 查询路由策略：规则匹配 vs 模型分类

**决策**: 规则匹配 + 轻量模型混合

**理由**:
- 规则匹配处理明确场景（如包含"对比"、"列出"等关键词）
- 轻量模型（如逻辑回归）处理模糊场景
- 性能好，延迟低

### 5. 配置驱动 vs 代码硬编码

**决策**: 配置驱动

**理由**:
- 支持动态调整 Agent 工具和策略
- 无需代码修改即可切换检索模式
- 便于 A/B 测试和灰度发布

### 6. QueryRewriter 整合

**决策**: 将 QueryRewriter 功能整合到 query_understanding 节点

**理由**:
- QueryRewriter 职责是查询理解，与 query_understanding 一致
- 避免独立服务调用开销
- 代码更简洁，逻辑集中

### 7. Small-to-Big 定位

**决策**: Small-to-Big 是基础检索策略，不是独立工具

**理由**:
- 文档向量化时已切分成小块（ChromaDB）和大块（ParentStore）
- 检索时使用小块检索 + 大块返回是默认行为
- 不是"可选工具"，而是"检索参数"

```
Small-to-Big 架构

索引阶段：
  原文 → 大块分割 → 小块分割 → 小块向量化 + 大块存储

检索阶段：
  查询 → 小块向量检索 → 查映射表 → 返回大块
         ↑                      ↑
    (ChromaDB)           (ParentDocumentStore)
```

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| langgraph 增加依赖复杂性 | 中等 | 锁定版本，添加依赖说明文档（langgraph 是独立包，不包含在 langchain 中） |
| Agent 检索延迟增加 | 高 | 设置超时，充分测试性能 |
| 代码重构引入回归 bug | 高 | 完整测试覆盖，通过 Git 版本控制快速回滚 |
| 配置错误导致运行时失败 | 中等 | 配置校验，默认值兜底 |

## Migration Plan

1. **Phase 1**: 创建新模块结构 (`rag/core/`, `rag/tools/`, `rag/agents/`)
2. **Phase 2**: 实现新架构功能，添加单元测试
3. **Phase 3**: 删除旧代码 (`rag/rag_utils.py` 等)，切换到新架构
4. **Phase 4**: 验证功能正常，提交代码

**回滚策略**: 通过 Git 版本控制回滚到上一提交，而非保留旧代码并行运行
