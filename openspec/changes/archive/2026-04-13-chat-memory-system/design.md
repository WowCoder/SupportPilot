## Context

当前 SupportPilot 系统已有 RAG 向量数据库基础设施（ChromaDB + ParentDocumentStore），支持文档上传、分块、检索功能。客服聊天场景需要：
1. 与 LLM 对话时保持上下文连贯性
2. 避免长对话导致 token 超限
3. 从历史对话中沉淀知识到向量库

### 现有基础设施
- `rag/rag_utils.py`: 向量检索工具
- `rag/parent_store.py`: 大块文档存储
- `app/models`: SQLAlchemy 数据模型
- `app/api/routes.py`: REST API 路由

## Goals / Non-Goals

**Goals:**
- 实现滑动窗口记忆机制，保留最近 5 条完整聊天记录（可配置）
- 超出窗口的记录标记为"待压缩"，累积 3-5 条后批量异步压缩
- 关单后可选生成 FAQ，直接录入现有向量数据库
- 检索时支持：窗口内记录 + 历史摘要 + FAQ 向量检索
- Query Rewrite：检索前改写用户 query，融入对话历史上下文

**Non-Goals:**
- 不实现实时多端同步（单会话单实例）
- 不实现语音/图片记忆（仅文本）
- 不修改现有 RAG 分块逻辑（FAQ 作为独立文档源）

## Decisions

### 1. 存储结构：关系型 + 向量混合存储
**决策**：使用 SQLite 存储原始聊天记录和摘要，FAQ 复用现有 ChromaDB

**理由**：
- SQLite 已在使用，支持窗口查询（ORDER BY + LIMIT）
- 摘要属于元数据，不需要向量检索
- FAQ 复用现有 ChromaDB collection，自动享受 Small-to-Big、混合搜索、重排序
- 通过 `metadata.source = "faq_from_session"` 区分 FAQ 和普通文档

**替代方案**：独立 FAQ collection
**未采用原因**：增加系统复杂度，且无法复用现有的 RAG 增强功能

### 2. 摘要触发时机：批量压缩 + 异步后台

**决策**：采用以下组合策略：
- **标记待压缩**：窗口超限时，仅将最旧记录标记为"待压缩"，不立即调用 LLM
- **批量压缩**：累积 3-5 条待压缩记录后，批量调用一次 LLM 生成摘要
- **异步处理**：压缩任务放入后台队列，不阻塞用户发送消息
- **空闲时触发**：会话空闲超过 30 秒 或 待压缩记录达 5 条时，触发批量压缩
- **兜底机制**：无论队列状态如何，超过 2 分钟强制触发一次压缩

**理由**：
- 批量压缩减少 LLM 调用次数（5 条一起压缩 vs 每条调用一次）
- 异步处理不影响用户发送消息的体验
- 空闲时触发避免与工作流竞争资源
- 兜底机制防止长对话场景下队列永不触发

**替代方案对比**：
| 方案 | Token 效率 | 延迟影响 | 实现复杂度 |
|------|-----------|---------|-----------|
| 实时压缩（每条触发） | 差 | 高 | 低 |
| 批量异步压缩 | 优 | 低 | 中 |
| 定时任务（每日） | 最优 | 无 | 低 |

**未采用纯定时任务原因**：长对话场景下，窗口内会堆积过多完整记录，失去压缩意义

### 3. Query Rewrite 机制

**决策**：在检索前对用户 query 进行改写，融入对话历史上下文

**理由**：
- 用户的最新 query 可能包含代词（"它"、"这个"），需要结合历史才能理解
- Query rewrite 能让 RAG 检索更精准，提升 AI 回复质量
- 利用窗口内 5 条完整记录作为上下文

**改写策略**：
```
原始 query: "它支持哪些功能？"
窗口历史：[用户：产品 A 的数据导出功能，AI：支持 CSV 和 Excel...]
改写后：  "产品 A 的数据导出功能支持哪些格式？"
```

**实现方式**：
- 轻量级 LLM 调用（专门用于 rewrite，非完整回复生成）
- 或使用规则 + 关键词替换（更快，但质量略低）

### 4. FAQ 生成内容来源

**决策**：窗口记录（最新细节）+ 历史摘要（早期背景）

**理由**：
- 窗口内 5 条保留最新、最详细的对话内容
- 历史摘要提供早期对话的背景信息
- 两者结合既完整又控制 token 消耗
- 避免直接处理完整对话历史导致的 token 超限

**FAQ 生成流程**：
```
关单触发
    │
    ▼
获取窗口内 5 条完整记录
    │
    ▼
获取该会话所有历史摘要
    │
    ▼
拼接 → LLM 总结 → 提取 Q&A 对
    │
    ▼
相似度去重 → 入库 ChromaDB
```

### 5. 检索优先级：窗口记录 + 历史摘要 + FAQ

**决策**：AI 回复时的上下文拼接策略：

```
最终 context = 
    [窗口内完整记录]      -- 最新上下文，必选
  + [历史摘要 (可选)]      -- 早期背景，如果窗口内信息不足
  + [FAQ 检索结果]         -- 相关知识，通过 RAG 检索
```

**理由**：
- 窗口内记录最相关（最新上下文）
- FAQ 是沉淀的高质量知识
- 历史摘要作为补充背景

## Risks / Trade-offs

| Risk | Mitigation |
|------|------------|
| 摘要质量不稳定导致信息丢失 | 保留原始记录，摘要仅作为补充；提供人工修正接口 |
| FAQ 污染向量库（低质量重复） | 入库前做相似度去重（与现有 FAQ 相似度>0.9 则跳过） |
| LLM API 失败导致关单流程阻塞 | 异步重试机制：关单成功，FAQ 生成失败则加入重试队列 |
| 窗口大小配置不当 | 默认 5 条，支持动态配置（`app/config.py` 中调整） |
| 批量压缩队列堆积 | 设置队列上限（10 条），超限后降级为实时压缩 |
| 异步任务丢失 | SQLite 持久化待压缩队列，服务重启后可恢复 |
| Query Rewrite 增加延迟 | 使用轻量级模型，设置超时；失败时降级为原始 query |

## Migration Plan

1. **数据迁移**：无需迁移历史数据（从新关单开始累积）
2. **部署顺序**：
   - 第一步：创建数据库表（`chat_memory`, `faq_entries`）
   - 第二步：部署服务层代码
   - 第三步：部署 API 端点
3. **回滚**：删除新表，恢复关单流程（不影响现有功能）

## Open Questions

- 窗口大小默认值：5 条是否足够？（需观察实际对话长度）
- 摘要保留策略：是否需要多级摘要（最近 10 条摘要、更早摘要）？
- FAQ 审核流程：是否需要人工审核后再入库？
- 批量压缩阈值：3 条还是 5 条触发一次？（默认 5 条）
- 空闲时间阈值：30 秒是否合适？（需实测）
- 兜底时间阈值：2 分钟是否合适？（需实测）
- Query Rewrite 使用轻量 LLM 还是规则引擎？（推荐轻量 LLM）
