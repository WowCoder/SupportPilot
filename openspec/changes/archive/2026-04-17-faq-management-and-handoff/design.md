## Context

SupportPilot 当前使用 Agentic RAG 系统进行智能问答，但缺少完整的工单生命周期管理和 FAQ 审核机制。现有聊天记忆系统（chat-memory）独立运行，需要在此基础上增加轮次计数、工单状态、FAQ 审核流程。

**现有架构：**
- 聊天记忆：`app/services/chat_memory.py` - 短期窗口 + 批量压缩
- FAQ 生成：`app/services/faq_generator.py` - 自动生成 FAQ
- RAG 检索：`rag/service.py` - 向量检索

**约束条件：**
- 保持现有聊天记忆系统独立运行
- FAQ 向量化使用现有 ChromaDB 和 Small-to-Big 检索
- 前端基于现有模板扩展

## Goals / Non-Goals

**Goals:**
- 实现对话轮次计数和人工介入触发
- 实现工单关闭功能
- 实现 FAQ 审核工作流（AI 生成→人工修改→确认→向量化）
- 实现 FAQ 管理后台（CRUD + 向量数据库同步）

**Non-Goals:**
- 不改变现有 RAG 检索核心逻辑
- 不重构聊天记忆压缩机制
- 不涉及多租户或权限系统变更

## Decisions

### 1. 工单状态管理

**决策：** 使用独立工单表 `support_ticket`，而非在 session 中存储状态

**理由：**
- 支持跨会话追踪（用户刷新页面后状态保留）
- 便于统计和历史追溯
- 技术支持可查看工单详情

**替代方案：** 使用 Redis 存储会话状态
**未采用原因：** 数据持久化需求、历史分析需求

### 2. 人工介入触发时机

**决策：** 固定 3 轮后触发（可配置）

**理由：** 
- 3 轮是行业常见阈值（给予 AI 充分机会，又不过长）
- 简单明确，用户容易理解

**替代方案：** 基于 AI 判断（如连续低置信度）
**未采用原因：** 实现复杂、可解释性差

### 3. FAQ 存储设计

**决策：** FAQ 审核表独立存储，与向量数据库分离

**理由：**
- 支持审核状态追踪（待审核/已确认/已拒绝）
- 支持版本历史（修改记录）
- 向量数据库仅存储已确认的 FAQ

**数据结构：**
```sql
faq_entries: (id, question, answer, category, status, created_by, confirmed_by, vector_ids[])
faq_versions: (id, faq_id, question, answer, change_reason, changed_by, created_at)
```

### 4. FAQ 向量化策略

**决策：** 使用 Small-to-Big 检索，FAQ 作为独立文档源

**理由：**
- 与现有文档检索保持一致
- FAQ 通常为短文本，适合 child chunk 大小
- 支持来源追溯（返回时知道来自 FAQ）

### 5. 实时同步 vs 批量同步

**决策：** 实时同步（FAQ 增删改时立即更新向量库）

**理由：**
- 实现简单
- 数据一致性高
- FAQ 操作频率低，性能影响可忽略

## Risks / Trade-offs

| Risk | Mitigation |
|------|------------|
| 3 轮阈值可能不适合所有场景 | 设计为配置项，后续可按需调整 |
| FAQ 审核流程可能增加技术支持负担 | 提供一键接受 AI 建议，减少操作步骤 |
| 向量数据库同步失败导致数据不一致 | 添加事务回滚机制和错误日志 |
| 工单表增长过快 | 定期归档历史工单，添加索引优化查询 |
| 人工介入后无技术支持响应 | 添加超时提醒、邮件通知机制（后续迭代） |

## Migration Plan

1. **数据库迁移**：执行 SQL 脚本创建工单表、FAQ 表
2. **后端服务**：部署新 API 路由
3. **前端页面**：更新聊天界面、新增 FAQ 管理页
4. **FAQ 初始化**：现有 FAQ 文档迁移到新表（如有）

**回滚策略：**
- 保留旧版代码分支
- 数据库表可回滚删除
- 不影响现有 RAG 检索功能
