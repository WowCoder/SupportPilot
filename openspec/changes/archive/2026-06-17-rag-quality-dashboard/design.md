## Context

当前系统 RAG 检索过程完全黑盒，调优依赖人工在测试页抽检。需要三个维度来让检索质量可量化、可观察：用户真实反馈、检索过程日志、LLM 自动评估。项目已有 SQLite、Flask-Login、LLM 客户端等基础设施，不需要引入新依赖。

## Goals / Non-Goals

**Goals:**
- 用户在 AI 回复下点赞/踩，数据持久化
- 每次 RAG 检索自动记录完整日志（查询、结果、分数、耗时）
- LLM-as-Judge 对检索结果打分（1-5 分，评估相关性/完整性）
- 为技术支持提供质量 Dashboard 页面，包含统计、日志列表、检索过程可视化
- 零新增 Python 依赖，复用现有 llm_client

**Non-Goals:**
- 不做自动化告警/通知
- 不做 A/B 测试框架
- 不做反馈自动触发重新检索

## Decisions

### 1. SQLite 存储，两张表
**选择**: `rag_retrieval_logs` + `user_feedback` 两张表，SQLite
**原因**: 项目用 SQLite，无需新增数据库。两张表小而独立，查询简单。每天最多几百条，不存在性能瓶颈。
**替代方案**: ChromaDB 存日志 — 放弃，和向量检索混在一起不清晰。

### 2. 检索日志在 rag_service.retrieve() 内置记录
**选择**: 在 `RAGService.retrieve()` 方法末尾自动记录日志，不需要各调用方手动记录
**原因**: 统一入口，覆盖所有检索调用（聊天、测试页、API）不遗漏。打开 `enable_logging=True` 开关控制。
**替代方案**: 各调用方自行记录 — 容易遗漏，维护成本高。

### 3. LLM-as-Judge 异步触发，不阻塞检索
**选择**: judge 评估在检索完成后异步调用，不计入检索耗时。Dashboard 页提供 "重新评分" 按钮手动触发
**原因**: LLM 调用耗时 2-5 秒，不能拖慢用户聊天体验。首次检索记录日志后，可以延迟触发或手动触发评分。
**替代方案**: 同步评分 — 用户发消息多等 3 秒，不可接受。

### 4. LLM-as-Judge prompt 设计
**选择**: 一次性让 LLM 从 3 个维度打分（relevance/completeness/noise），1-5 分
**原因**: 信息量比单一分数丰富，便于定位问题。"噪声分"直接对应 FanFan 你说的"搜到无关内容"。
**替代方案**: 只打分不加理由 — 可解释性差。

### 5. Dashboard UI 复用现有设计系统
**选择**: 使用现有 topnav + page-content + card + data-table 组件
**原因**: 和项目其他页面风格一致，无需写新 CSS。
**替代方案**: 做独立的可视化页面 — 过度设计。

## Risks / Trade-offs

- **LLM-as-Judge 本身有幻觉风险**: LLM 评估 LLM 的检索结果，可能不准确。缓解：Judge 打分同时输出理由，技术支持可以人工复核。
- **日志表可能膨胀**: 生产环境日均 100+ 检索，一个月 3000 条。缓解：SQLite 足够，Dashboard 分页展示。未来可加自动清理策略。
- **反馈按钮可能被滥用**: 用户乱点。缓解：只记录不干预系统行为，Dashboard 做统计分析时自然会排除异常值。
